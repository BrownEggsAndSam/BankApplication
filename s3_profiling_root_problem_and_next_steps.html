S3 Profiling: Root Problem, Investigation Plan, and Next Steps

This page explains what the developer actually identified, why the current S3 automation requirement is incomplete, what questions still need answers, and what evidence should be collected before the team commits to a solution or timeline.

1. What happened in the meeting?

The team originally discussed S3 profiling as though it could follow a simple path:

Read S3 Inventory
→
Find DSET
→
Send one path to Pentaho
→
Ingest recent files
→
Profile and create collection
The developer identified that this happy path is not consistently valid. The same DSET may appear under multiple nested directories, dates, partitions, archives, or application-specific folder structures. Therefore, identifying a DSET does not automatically produce one unique ingestion path.

2. The real root problem

The current design assumes this relationship:

Dataset ID → One S3 directory → One recent population

But the inventory may actually look like this:

Dataset ID
  ├── load_date=2026-06-01/dataset_615/
  ├── load_date=2026-06-02/dataset_615/
  ├── archive/2026-05/dataset_615/
  └── application/table/date/dataset_615/

This creates a one-to-many relationship between a dataset and its S3 paths. The automation does not yet know which path or paths are valid, current, non-duplicate, and appropriate for profiling.

This is not only a coding defect. It is primarily a requirements and solution-design gap. Development cannot implement reliable logic until the expected behavior is defined.

3. Questions the developer needs answered

  • How is the DSET derived when folder structures are different?
  • Is the DSET always the right-most matching folder?
  • Can one DSET appear under several dates or branches?
  • Should all valid paths be ingested or only the newest one?
  • How should archive, manifest, log, and zero-byte objects be handled?
  • What happens when multiple homogeneous file groups are found?
  • Which resource ID is needed for profiling, and how is it retrieved?
  • Does the required API work in production, not only test?

4. Why the existing S3 Inventory is helpful but not enough

The S3 Inventory should be the starting point because it lets the team inspect bucket contents without asking Pentaho to discover everything. However, the inventory still needs interpretation logic.

Inventory gives us Still needs to be determined
Bucket and object key Which portion of the key represents the authoritative dataset path
Repeated DSET occurrences Whether they are duplicates, partitions, dated copies, archives, or separate valid populations
File modification dates Whether date filtering should use object date, folder date, receive date, load date, or another business date
File extensions and sizes Which formats are supported and which files should be excluded
Full object keys Which folder level should be submitted to the ingestion API

5. What Nach should provide

Nach's examples are important because they can prove the non-happy-path structures and help define the resolution rules.

  • One straightforward example where one DSET maps to one clear path.
  • One example where the same DSET appears under multiple dates.
  • One example with several nested folders before the DSET.
  • One example containing archive, manifest, log, or temporary files.
  • One example where the dataset is not the final folder.
  • One example where multiple CSV structures exist for the same DSET.
  • The expected correct ingestion path for each example.
  • The expected file population for each example.
The best output from Nach would be a small annotated spreadsheet containing: bucket, full key, identified DSET, expected parent path, include/exclude decision, and the reason.

Reuse the orchestration pattern, but document every parameter or metadata step that differs for S3.

8. Suggested investigation sequence

  1. Collect 5–10 representative S3 Inventory examples. Include both happy-path and difficult folder structures.
  2. Annotate the expected answer. For every example, identify the DSET, valid path, date population, exclusions, and expected collection result.
  3. Build a dataset-to-path rule. Decide whether the output is one path or multiple paths per DSET.
  4. Review Swagger. Confirm whether the ingestion API supports date filtering, recursion, explicit files, wildcards, and job status.
  5. Run API tests in test and production. Do not assume production behavior matches test.
  6. Run GraphQL tests. Confirm how to obtain resource IDs, dictionary IDs, pattern IDs, and collection metadata.
  7. Document S3-specific profiling inputs. Header handling, delimiters, patterns, dictionaries, document processing, and file sampling.
  8. Define collection rules. Decide what to do when there are multiple file structures or multiple successful profiling groups.
  9. Separate internal work from vendor blockers. Only escalate items the team cannot solve with inventory, REST, GraphQL, or internal mapping logic.

This is likely better than adding one full-path column to the existing dataset compliance table, because one dataset may require multiple path records.

10. Vendor dependency: what is truly mandatory?

The team should avoid asking the vendor to solve internal Fannie Mae dataset-identification logic. However, some platform capabilities may still be unavoidable.

Capability Can likely be solved internally? Vendor needed?
Identify DSET from S3 Inventory Yes, with rules and examples No
Map one DSET to multiple paths Yes, with a child mapping table No
Exclude unsupported files Yes, before submission No
Limit to recent objects before submission Possibly, if explicit paths/files can be generated Maybe
API date-range filtering No, if API only accepts a broad directory Likely mandatory
Wildcard-path support Can sometimes be avoided with internal path expansion Helpful, but not always mandatory
Production API defects No Mandatory

11. Questions to answer before grooming is complete

  • What is the authoritative rule for finding the DSET in an S3 key?
  • Can the same DSET have multiple valid ingestion paths?
  • Which date field determines the 30-day population?
  • Should date filtering happen internally or inside Pentaho?
  • Can the API accept an explicit list of files or only a directory?
  • Does the API recursively process subfolders?
  • Does wildcard support exist in REST, GraphQL, or neither?
  • How are entity/resource IDs retrieved after ingestion?
  • Which S3 profiling inputs differ from DB profiling?
  • How are multiple homogeneous groups converted into collections?
  • What works in test but fails in production?
  • Which September commitments depend on unresolved vendor work?

12. One-paragraph summary for leadership

The developer identified that the S3 profiling requirement assumes a dataset can be mapped to one unique folder path, but the S3 Inventory contains inconsistent and nested structures where the same DSET may appear under multiple dates and directory branches. Before automation can be built, the team must define how to derive the authoritative DSET path, how to handle multiple paths and file populations, and how collections should be created from multiple profiling results. S3 Inventory can reduce vendor dependency, but Pentaho API date filtering and production API reliability may still be mandatory blockers.
